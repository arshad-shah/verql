Plugin System & Extensibility

How community plugins extend the app — the VS Code model adapted for a database client

A

npm-Based Plugin Registry

Plugins are npm packages with a manifest. Installed via a built-in marketplace UI. Each plugin runs in a sandboxed iframe or web worker for security. Plugins can add:

  • Activity bar panels — their own sidebar view (like the VS Code model)
  • Database drivers — add support for new databases (Redis, Cassandra, etc.)
  • Result visualizations — custom chart types, data renderers
  • Themes — custom color schemes and icon packs
  • Query transformers — SQL formatters, linters, migration generators
  • Import/export formats — Parquet, Excel, Avro, custom formats
  • Keybinding presets — DataGrip-style, DBeaver-style, etc.

Pros

  • Familiar to JS/TS developers
  • npm infrastructure handles versioning, publishing
  • Sandboxed execution = safe third-party code
  • Hot-reload during development

Cons

  • npm registry is noisy — need curation/quality control
  • Sandboxing limits what plugins can do
B

Custom Registry + Plugin SDK

Your own plugin registry (like VS Code Marketplace) with a TypeScript SDK. Plugins must conform to a typed API. Reviewed/verified plugins get a badge. Same extension points as Option A, but with tighter control.

Pros

  • Full control over quality and security
  • Typed SDK = better developer experience
  • Verified badge builds trust
  • Can evolve API independently of npm

Cons

  • Must build and maintain the registry infrastructure
  • Higher barrier to entry for plugin authors
  • Slower ecosystem growth initially

💡 Recommendation: Option A — npm-based, evolve to B later

Start with npm packages + a typed Plugin SDK. Use a contributes manifest (like VS Code's package.json extensions) to declare what a plugin provides. Ship an in-app "Extensions" panel in the activity bar for discovery and install. You can add a curated registry later once the ecosystem grows — but npm gets you bootstrapped immediately with zero infrastructure.
